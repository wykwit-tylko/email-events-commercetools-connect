# Proxy fans out signed Email Events to multiple publishers

`OUTBOUND_PUBLISHER_CONFIG` now accepts a JSON array of publisher configs (a single object is still accepted and normalized to a one-element array). A `CompositePublisher` forwards each Commerce Notification to every configured publisher in parallel; if any publisher fails, the inbound request fails so commercetools redelivers, and every consumer is expected to be idempotent. This supersedes the "without designing fan-out before it is needed" clause of ADR 0001: a second consumer (the shelfmarket store backend) now needs the same Email Events.

A second publisher type, `http-webhook`, POSTs each Commerce Notification as a signed Email Event to an HTTP endpoint, adapting only to the endpoint's wire format (a JSON body plus signature and replay headers) without interpreting the notification. This is how the proxy delivers to the shelfmarket store backend, which has no message-queue consumer.

Delivery is authenticated and replay-protected. The publisher computes `HMAC-SHA256(secret, "webhook-v1\n{timestamp}\n{messageId}\n{body}")` over the exact request body bytes and sends the signature, timestamp, and message id as headers. The receiver verifies the HMAC in constant time, rejects timestamps outside a 5-minute window, and deduplicates by message id within that window. The shared secret is a dedicated `emailEventSecret` (`emailEventSecret` in the publisher config, `NUXT_EMAIL_EVENT_SECRET` on the receiver) — not `ORDER_LINK_SECRET`, which is shared with the storefront for a different purpose and would let the storefront forge events if reused here. A pinned known-answer vector on both sides keeps the cross-runtime contract (WebCrypto on the proxy, Node `createHmac` on the receiver) from drifting.
