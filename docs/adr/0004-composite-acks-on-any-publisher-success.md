# Composite Publisher acks commercetools on any single publisher success

The `CompositePublisher` now resolves the inbound request as soon as any one Outbound Publisher accepts the Commerce Notification, and rejects (so commercetools redelivers) only when every configured publisher fails. This supersedes the "if any publisher fails, the inbound request fails so commercetools redelivers" clause of ADR 0003, which held the durable Cloudflare Queue publisher hostage to the fragile `http-webhook` publisher: when the webhook endpoint was slow or down, the queue leg had already accepted the Email Event, yet the proxy returned non-2xx, so commercetools redelivered and the queue leg accepted the same Commerce Notification again on every retry. The email worker's dedupe then suppressed the duplicates correctly but burned throughput on a hot loop, with no additional email ever sent or lost.

Because redelivery could not help — it only re-enqueued the notification to whichever publishers had already succeeded — the proxy now prefers one confirmed delivery over a futile retry. Each individual publisher failure is still logged at warn level as `outbound publisher failed` (with its index) so a degraded publisher is never silenced, and monitoring is expected to alert on the sustained rate of those warnings for any single publisher index rather than on commercetools redelivery. The `publisherIndex` in every log and in the all-failures error message corresponds to the position in `OUTBOUND_PUBLISHER_CONFIG`, and failures are reported in index order so operators can map them back to the configuration. Every consumer remains idempotent: a slow publisher can still succeed in the background after the composite has already resolved on another publisher's success, which is safe because the Cloudflare Queue dedupes via `sent:${notificationId}` and the webhook receiver dedupes by signed message id within its replay window.
